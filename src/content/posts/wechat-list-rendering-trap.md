> 你的 HTML 在浏览器里完美，在微信里就多出几行空白。这不是你的 CSS 写错了——是微信的渲染引擎在做手脚。

## 现象

通过 API 上传公众号草稿，文章里有一段普通的无序列表：

```html
<ul>
  <li>第一条内容</li>
  <li>第二条内容</li>
  <li>第三条内容</li>
</ul>
```

预览效果：每个 `<li>` 之间出现了多余的空白行，项目符号位置也有偏移。在桌面浏览器预览正常，但微信客户端内阅读时问题明显。

## 根因

微信公众号的文章渲染器会注入自己的内置样式，覆盖你通过 inline style 设置的值。关键的干扰项：

- `li { margin-bottom }` — 微信给每个列表项加了下边距
- `ul { margin: 0 0 16px 0 }` — 列表块本身也有额外间距
- 某些情况下 `<li>` 之间的空白文本节点也会被渲染为可见间距

即使你在模板里把这些值全部清零：

```css
ul, ol { margin: 0; padding: 0; }
li { margin-bottom: 0; }
```

premailer inline 化后上传，微信仍然可能用自己的样式覆盖。这取决于微信客户端版本和渲染路径。

## 解法

**放弃 `<ul><li>` 标签，用纯文本模拟列表。**

在 Markdown 源文件中，不写列表语法：

```markdown
• 第一条内容

• 第二条内容

• 第三条内容
```

每个 bullet 之间空一行，让转换器将每条生成为独立的 `<p>` 标签。`<p>` 的间距行为在微信中比 `<li>` 可控得多。

最终 HTML 结构：

```html
<p>• 第一条内容</p>
<p>• 第二条内容</p>
<p>• 第三条内容</p>
```

## 防御性措施

即使改用 `<p>` 方案，模板 CSS 中仍保留 `ul/ol/li` 的清零规则：

```css
ul, ol { margin: 0; padding: 0; list-style: none; }
li { margin: 0; padding: 0; }
```

这样即使某篇文章不小心用了列表语法，至少不会出现特别夸张的间距问题。

## 要点

1. **不要信任浏览器预览** — 微信客户端的渲染和桌面浏览器不一样
2. **inline style 不是万能的** — 微信可以覆盖你的 inline 样式
3. **`<p>` 比 `<li>` 可控** — 微信对段落标签的干预比列表标签少
4. **在真机预览验证** — 每次模板改动后，必须在手机微信里查看效果

这个坑不大，但如果你是通过 API 自动化发布公众号文章，会反复踩到。记录一下，省得下次排查半天。
